# GUIDE

AI 협업 시 참고해야 할 작업 규칙과 가이드.

---

## 공통 규칙

### ⚡ 마커

사용자가 문서 어디에든 ⚡를 넣으면 **사용자의 의견/코멘트**를 표시한 것이다.

- AI는 ⚡가 포함된 내용을 읽고 문서에 반영해야 함
- 반영 후 ⚡ 마커와 해당 코멘트는 삭제

### 완료 기준 (Criteria)

`plan.md`와 `stages/` 문서 작성 시, **각 섹션 끝에 완료 기준**을 명시한다.

```markdown
### 1.1 사용자 인증

- OAuth 연동
- 세션 관리

**완료 기준:**
- [ ] Google OAuth 로그인 성공
- [ ] 로그아웃 시 세션 삭제 확인
```

이렇게 하면 O/X로 완료 여부를 판단할 수 있다.

## 작업 시작 시 읽는 순서

1. `plan.md` → 프로젝트 방향과 현재 stage 확인
2. 현재 진행 중인 `stages/stageX.md`
3. 필요시 `inbox.md` (사용자 요청 시)

---

## 폴더 구조

| 폴더/파일 | 용도 |
|-----------|------|
| `plan.md` | 프로젝트 방향, 의도, 전체 로드맵, **현재 상태** |
| `inbox.md` | 사용자의 수신함 (아이디어, 피드백, 메모) |
| `specs/` | 확정된 규칙/스펙 (아키텍처, 코드 컨벤션, 디자인 토큰 등) |
| `decisions/` | 결정 히스토리 (왜 이렇게 결정했는가) |
| `stages/` | 각 stage별 세부 계획 |
| `devlog/` | 일일 개발 로그 (`__template__.md` 참고) |

## 작업 규칙

### inbox.md 처리

- **inbox는 사용자의 수신함이다** — 사용자가 자유롭게 메모를 쌓는 곳
- 사용자가 "수신함 검토해줘" 요청 시:
  1. `inbox.md`를 진입점으로 읽기 (처리 지침이 inbox.md에 포함되어 있음)
  2. inbox.md의 지침에 따라 처리

### specs/ 작성

- 확정된 규칙/스펙을 기록하는 곳
- 파일명: 주제 기반 (예: `architecture.md`, `code-convention.md`, `design-tokens.md`)
- `## 헤딩`, `### 헤딩`은 CLAUDE.md 인덱스에 TOC로 노출됨

### decisions/ 작성

- 결정의 히스토리/배경을 기록하는 곳
- 파일명: `YYMMDD-주제.md` (예: `241203-css-tokens.md`)
- 내용 구조:
  ```
  # 제목

  ## 배경
  왜 이 결정이 필요했는가

  ## 결정
  무엇을 어떻게 하기로 했는가

  ## 대안 검토 (선택)
  고려했지만 선택하지 않은 옵션들
  ```

### stages/ 작성

- 파일명: `stageN_제목.md` (예: `stage1_초기설정.md`, `stage2_핵심기능.md`)
- `plan.md`의 Stage 개요와 연결
- 세부 작업 항목은 체크박스로 관리
- `## 헤딩`은 CLAUDE.md 인덱스에 TOC로 노출됨

### devlog/ 작성

- 파일명: `YYMMDD.md` (예: `241203.md`)
- 구조:
  ```
  ## 해결되지 않은 문제
  - 문제:
  - 예상 원인:
  - 시도해본 것:
  - 다음에 시도할 것:

  ## 오늘 한 일
  - [ ] stage1.1 - 작업 내용
  - [x] stage1.2 - 완료된 작업
  ```
